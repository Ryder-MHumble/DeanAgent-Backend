# 各项目演进路线与职责划分

## 1. 文档目标

这份文档关注具体项目，不重复平台总纲，重点回答：

1. 每个项目接下来该做什么
2. 每个项目不该再做什么
3. 各项目之间未来如何协同而不是互相侵蚀边界

---

## 2. 总体分工

| 项目 | 未来定位 | 核心职责 | 主要风险 |
|---|---|---|---|
| `DeanAgent-Backend` | `情报引擎 Core` | 底座、模型、处理、服务、治理 | 继续业务特化、边界膨胀 |
| `Dean-Agent-Fronted` | 领导驾驶舱 | 态势消费、重点提醒、决策辅助 | 功能泛化为万能门户 |
| `Scholars-System` | 知识运营台 | 结构化维护、审核、校准、运营 | 被当作简单展示前台 |
| `academic-monitor` | 统一监测框架来源 | 目标监测抽象、规则与告警能力 | 与 Core 双轨并行 |
| `Dean-NanoBot` | 统一 Agent 入口 | 问答、工具调用、订阅、推送 | 绕开治理直接做系统写入 |

---

## 3. `DeanAgent-Backend` 演进路线

## 3.1 目标定位

统一升级为 `情报引擎 Core`。

## 3.2 未来 3 个阶段

### 阶段 A：底座收口

重点：

- 统一对象模型
- 统一信源与 API 口径
- 建立数据分层
- 提炼公共能力

### 阶段 B：平台化

重点：

- 引入统一监测框架
- 引入统一通知与订阅
- 引入统一工具层
- 引入统一权限与审计

### 阶段 C：稳定交付

重点：

- 为多个前台和 Agent 稳定供给
- 底层重构不影响上层
- 输出平台指标和治理指标

## 3.3 不该继续做的事

- 不再把前台专属逻辑直接编码进底座 API
- 不再让每个业务模块各自长出一套数据模型
- 不再忽视血缘、口径、审计等治理能力

---

## 4. `Dean-Agent-Fronted` 演进路线

## 4.1 目标定位

收口为：

**领导决策驾驶舱**

## 4.2 未来 3 个阶段

### 阶段 A：去 Mock、接真实能力

重点：

- 与 Core 的简报、政策、人事、科技、高校数据打通
- 建立统一刷新、状态、来源、可信度展示

### 阶段 B：从“信息展示”升级为“判断与行动”

重点：

- 增加重点判断卡片
- 增加机会 / 风险 / 建议输出
- 增加关注对象和提醒机制

### 阶段 C：收口边界

重点：

- 将运营型模块下沉
- 与 Portal、NanoBot、任务中心联动

## 4.3 不该继续做的事

- 继续扩张为管理后台
- 在领导前台做重运营表单和长流程
- 过早接入大量内部系统导致系统目标失焦

---

## 5. `Scholars-System` 演进路线

## 5.1 目标定位

升级为：

**学术生态知识运营台**

## 5.2 未来 3 个阶段

### 阶段 A：模型对齐

重点：

- 与 Core 的 scholar / institution / project / event 对齐
- 明确哪些字段来自采集，哪些字段来自人工维护

### 阶段 B：运营能力增强

重点：

- 导入、审核、纠错、比对
- 来源、更新时间、证据链、可信度展示
- 变更历史与人工校准留痕

### 阶段 C：图谱化

重点：

- 学者关系
- 机构关系
- 项目与活动关系
- 合作证据可视化

## 5.3 不该继续做的事

- 仅停留在“静态展示型学者数据库”
- 与 `Dean-Agent` 抢领导驾驶舱定位
- 与 `NanoBot` 抢入口定位

---

## 6. `academic-monitor` 演进路线

## 6.1 目标定位

作为统一监测框架的原型和能力来源，逐步并入 `情报引擎 Core`。

## 6.2 未来 3 个阶段

### 阶段 A：抽象梳理

重点：

- 提炼 `MonitoringTarget`
- 提炼监测策略、评估规则、通知机制
- 标记与 Core 重复的部分

### 阶段 B：能力迁移

重点：

- 将目标、规则、通知、存储逐步对齐到 Core
- 将独立调度与平台调度整合

### 阶段 C：工作台显化

重点：

- 形成 `Monitoring Center`
- 让监测能力成为平台标准能力，而不是孤立项目

## 6.3 不该继续做的事

- 再发展一套与 Core 平行的完整平台
- 自建独立身份、通知、存储、调度体系

---

## 7. `Dean-NanoBot` 演进路线

## 7.1 目标定位

升级为：

**统一 Agent 入口、工具编排层与消息触达层**

## 7.2 未来 3 个阶段

### 阶段 A：工具标准化

重点：

- 平台工具清单
- 工具 Schema
- 权限校验
- 调用审计

### 阶段 B：订阅与推送标准化

重点：

- 个人订阅
- 角色订阅
- 部门订阅
- 广播和群组能力

### 阶段 C：任务闭环

重点：

- 对话转任务
- 任务转通知
- 通知转回执
- 必要时跳转 GUI 完成操作

## 7.3 不该继续做的事

- 每个 skill 自己偷偷绑定后端接口
- 绕过权限直接写平台数据
- 让聊天入口承接复杂结构化运营流程

## 8. 项目之间的协同关系

## 8.1 推荐协同链路

```text
情报引擎 Core
  -> Scholars-System 负责知识维护
  -> Monitoring Center 负责目标监测
  -> Athena 负责深度分析
  -> Dean-Agent 负责领导消费
  -> NanoBot 负责自然语言访问和触达
```

## 8.2 典型协作场景

### 场景 1：发现新学者信息

- Core 采集与初步处理
- `Scholars-System` 做人工校准与补录
- `NanoBot` 可回答查询
- `Dean-Agent` 可消费高价值结果

### 场景 2：监测目标异常

- Core / 监测框架触发告警
- `Monitoring Center` 人工确认
- `NanoBot` 推送提醒
- `Athena` 可做专题延展分析

### 场景 3：领导关注某一专题

- `Dean-Agent` 展示摘要和判断
- `Athena` 提供深度分析
- `NanoBot` 支持临时追问

## 9. 2026-2027 排期建议

| 时间段 | 重点项目 | 核心目标 |
|---|---|---|
| 2026 Q2 | `DeanAgent-Backend` | 平台定义、对象模型、口径统一 |
| 2026 Q3 | `DeanAgent-Backend` + `academic-monitor` + `Dean-NanoBot` | 监测、订阅、工具层统一 |
| 2026 Q4 | `Dean-Agent-Fronted` + `Scholars-System` | 工作台边界收口与重构 |
| 2027 H1 | 全平台 | Agent 与 GUI 闭环、统一门户化 |

## 10. 成功标志

### `DeanAgent-Backend`

- 可以作为平台底座稳定支撑多个前台
- 关键口径统一
- 引入治理能力

### `Dean-Agent-Fronted`

- 真正服务领导使用
- 不再像万能门户

### `Scholars-System`

- 变成高价值知识运营工具
- 支撑数据可信度提升

### `academic-monitor`

- 抽象被保留
- 孤立性被消除

### `Dean-NanoBot`

- 成为可信的统一入口
- 不绕开平台治理

---

## 11. 一句话总结

各项目未来不是各自冲刺，而是：

**围绕一个 Core 做不同职责分工，最终形成协作链路而不是竞争关系。**
