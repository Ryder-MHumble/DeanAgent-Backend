# 产品矩阵与角色工作台设计

> 面向平台协作、产品规划和需求分流的工作文档  
> 更新日期：2026-03-31  
> 关联总纲：[情报引擎产品战略与演进路线图](./情报引擎产品战略与演进路线图.md)

---

## 1. 文档目标

这份文档回答 3 个问题：

1. 平台到底服务哪些角色
2. 每个角色应该使用什么产品形态
3. 新需求进入时，应该落到哪个工作台，而不是继续新建系统

---

## 2. 平台级产品矩阵

### 2.1 总体矩阵

| 层级 | 产品 / 能力 | 定位 | 核心用户 | 交互方式 |
|---|---|---|---|---|
| L0 | `情报引擎 Core` | 情报底座、知识底座、监测底座 | 平台内部 | API / Tools / Pipeline |
| L1 | `Portal / Workbench` | 统一门户、统一搜索、统一通知、统一任务 | 全院各角色 | GUI |
| L1 | `NanoBot` | 自然语言入口、消息触达、任务触发 | 全院员工 | Agent / IM |
| L2 | `Dean-Agent` | 领导决策驾驶舱 | 院长办、院领导 | GUI + Agent 辅助 |
| L2 | `Scholars-System` | 学术生态知识运营台 | 科研管理、知识运营 | GUI |
| L2 | `Athena` | 战略研判与专题分析台 | 战略发展、分析团队 | GUI + Agent 辅助 |
| L2 | `Monitoring Center` | 目标监测、规则评估、预警闭环 | 监测团队、专项治理团队 | GUI + Agent 辅助 |

### 2.2 一个关键原则

所有 L2 产品都不应该各自维护一套采集、存储、规则、身份、通知能力。

这些能力必须统一沉到 `情报引擎 Core` 和统一交付层，L2 产品只负责：

- 角色化视图
- 场景化流程
- 局部交互优化

## 3. 角色地图

### 3.1 主要角色

| 角色 | 主要目标 | 核心诉求 |
|---|---|---|
| 院领导 | 快速掌握态势并形成判断 | 高密度结论、风险、机会、建议 |
| 院长办 / 助理 | 做简报、排优先级、跟踪关键事项 | 汇总、提醒、行动建议、日程判断 |
| 科研管理 | 维护学者、机构、项目、活动、关系 | 结构化录入、台账治理、批量操作 |
| 战略发展 / 情报分析 | 做专题研判和趋势分析 | 深检索、多维筛选、专题输出 |
| 学术监测 / 审计人员 | 监测目标对象并追踪异常 | 规则配置、告警闭环、证据回溯 |
| 普通部门用户 | 快速提问和获得现成答案 | 低门槛访问、订阅和推送 |

### 3.2 角色与产品映射

| 角色 | 主产品 | 辅助产品 |
|---|---|---|
| 院领导 | `Dean-Agent` | `NanoBot` |
| 院长办 / 助理 | `Dean-Agent` | `NanoBot`、`Monitoring Center` |
| 科研管理 | `Scholars-System` | `NanoBot` |
| 战略发展 / 情报分析 | `Athena` | `NanoBot`、`Dean-Agent` |
| 学术监测 / 审计人员 | `Monitoring Center` | `NanoBot` |
| 普通部门用户 | `NanoBot` | `Portal` |

## 4. 工作台定位

## 4.1 `Dean-Agent`

### 角色定位

领导层的情报消费与决策辅助前台。

### 核心任务

- 每日简报
- 政策 / 人事 / 科技 / 高校态势感知
- 关键人、关键事、关键机会提醒
- 活动邀约与日程判断

### 不应持续扩张的方向

- 复杂台账维护
- 大量人工录入
- 学生事务全流程
- 项目督办系统化运营
- CRM 式人脉运营

这些能力更适合下沉到其他工作台或内部业务系统。

## 4.2 `Scholars-System`

### 角色定位

学术生态知识资产的运营后台。

### 核心任务

- 学者档案维护
- 机构树治理
- 项目与活动管理
- 合作关系、证据链补录
- 数据导入、审核、纠错、校准

### 本质

这是知识生产系统，不是情报消费系统。

## 4.3 `Athena`

### 角色定位

面向专题分析与深度研判的情报分析台。

### 核心任务

- 全量检索
- 多维过滤
- 专题分析
- 趋势追踪
- 对标与机会挖掘

### 本质

这是分析工作台，不是领导首页，也不是聊天机器人。

## 4.4 `Monitoring Center`

### 角色定位

目标监测与告警运营台。

### 核心任务

- 创建监测目标
- 配置规则与策略
- 查看告警与证据
- 人工确认与闭环
- 形成监测报告

### 本质

这是一个持续运行的监测控制面。

## 4.5 `NanoBot`

### 角色定位

全院统一自然语言入口。

### 核心任务

- 快速问答
- 快速检索
- 快速订阅
- 触发任务
- 接收推送

### 本质

这是平台的“操作面”和“消息触达面”，不是替代所有 GUI 的唯一前台。

## 5. GUI 与 Agent 的职责划分

### 5.1 适合放在 GUI 的需求

- 多表单、多字段录入
- 批量导入导出
- 多人协作
- 审批与审核
- 配置型工作
- 高密度对比分析

### 5.2 适合放在 Agent 的需求

- 一问一答式查询
- 跨系统取数
- 快速摘要和解释
- 订阅和提醒
- 快速触发一个已有流程

### 5.3 双轨协同规则

#### 规则 1：高频稳定工作流优先 GUI

例如：

- 学者数据维护
- 活动批量导入
- 机构树调整
- 审核和纠错

#### 规则 2：低频即时查询优先 Agent

例如：

- “最近有哪些北京 AI 政策值得关注？”
- “帮我查一下清华计算机系最近新增了哪些老师”
- “今天有哪些需要提醒领导的动态”

#### 规则 3：复杂任务由 Agent 拉起，由 GUI 完成

例如：

- Agent 先识别“需要补录某位学者资料”
- 再跳转 `Scholars-System` 完成补录
- 操作结果回流到通知和日志

## 6. 典型任务分发规则

### 6.1 需求分流表

| 需求类型 | 建议入口 | 最终落点 |
|---|---|---|
| 快速查找某个信息 | `NanoBot` | 直接回答或跳转工作台 |
| 浏览领导简报和态势 | `Dean-Agent` | 驾驶舱 |
| 维护学者 / 机构 / 活动台账 | `Scholars-System` | 知识运营台 |
| 做专题分析和数据研判 | `Athena` | 分析台 |
| 配置监测目标和预警规则 | `Monitoring Center` | 监测工作台 |
| 订阅某类事件推送 | `NanoBot` 或 `Portal` | 订阅中心 |

### 6.2 错误分流示例

以下情况应避免：

- 在 `Dean-Agent` 里堆大量后台管理页面
- 在 `NanoBot` 中承担复杂结构化录入
- 在 `Scholars-System` 中承接领导驾驶舱任务
- 为单一部门场景再新建一个同类前台

## 7. 角色工作台蓝图

### 7.1 推荐导航结构

```text
Portal
├── 我的首页
├── 全局搜索
├── 通知中心
├── 任务中心
├── 订阅中心
├── Dean-Agent
├── Scholars-System
├── Athena
└── Monitoring Center
```

### 7.2 公共能力应统一提供

以下能力不应在每个产品内重复建设：

- 统一登录与身份
- 全局搜索
- 收藏与关注
- 通知中心
- 订阅中心
- 任务中心
- 操作留痕

---

## 8. 未来新增产品的判断标准

只有同时满足以下条件，才值得考虑新增独立产品：

1. 目标用户与现有工作台明显不同
2. 工作流与现有工作台明显不同
3. 需要专门的信息架构和交互模型
4. 不能仅通过现有能力组合解决

否则，优先采用：

- 扩展现有工作台
- 新增角色页面
- 在 `Portal` 中增加入口
- 在 `NanoBot` 中增加工具与订阅

---

## 9. 当前最重要的产品决策

1. 明确 `Dean-Agent` 只做领导驾驶舱
2. 明确 `Scholars-System` 是知识运营台
3. 明确 `academic-monitor` 最终落位为 `Monitoring Center + 统一监测框架`
4. 明确 `NanoBot` 是入口层、工具层、触达层
5. 明确任何新需求先做分流判断，而不是先建新系统

---

## 10. 一句话总结

平台未来不是“很多项目并行”，而是：

**一套底座、一个统一入口、多个角色工作台。**
