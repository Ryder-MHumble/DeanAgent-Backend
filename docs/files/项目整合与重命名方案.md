# 项目整合与重命名方案

> 面向平台收口、对内沟通和后续仓库治理的工作文档  
> 更新日期：2026-03-31  
> 关联总纲：[情报引擎产品战略与演进路线图](./情报引擎产品战略与演进路线图.md)

---

## 1. 文档目标

这份文档回答两个现实问题：

1. 当前项目名称和边界是否还适合未来发展
2. 如果要收口成平台体系，应该如何整合和重命名

---

## 2. 当前命名存在的问题

### 2.1 项目名与真实角色不匹配

| 当前名称 | 实际角色 | 问题 |
|---|---|---|
| `DeanAgent-Backend` | 平台底座 | 名字像“院长系统后端”，但实际服务多个产品 |
| `Dean-Agent-Fronted` | 领导驾驶舱 | 名字拼写不统一，且弱化了 GUI 工作台属性 |
| `Scholars-System` | 知识运营台 | 名称过窄，容易被理解为简单学者库 |
| `academic-monitor` | 监测框架 | 更像技术项目名，未体现平台中的角色 |
| `Dean-NanoBot` | Agent 入口 | 名称仍强绑定 Dean 场景，不利于全院统一入口定位 |

### 2.2 命名方式偏历史，不利于平台叙事

当前命名更像：

- 历史阶段性的项目名
- 面向局部需求的定制系统名
- 技术实现导向的工程名

而不是：

- 平台底座
- 标准工作台
- 标准入口层

### 2.3 平台扩展时会持续增加认知成本

如果未来继续扩展而不处理命名：

- 新产品难以归类
- 老项目边界会持续模糊
- 新成员很难快速理解平台结构

---

## 3. 重命名原则

### 原则 1：按平台角色命名

名称首先表达“它在平台中干什么”，而不是“它最早是为谁做的”。

### 原则 2：按产品职责命名

名称应体现它是底座、工作台、入口还是能力域。

### 原则 3：中性优先

平台和底座名称应使用中性表达，避免继续绑定某个单一前台或单一历史项目。

### 原则 4：渐进替换

优先统一文档、定位和展示名称，再决定是否调整仓库名和部署名。

---

## 4. 推荐命名结构

### 4.1 平台总称

建议使用：

**研究院智能情报平台**

这是总称，适用于汇报、规划、产品矩阵和跨团队协作。

### 4.2 底座工作名

建议使用：

**情报引擎 Core**

这不是对外品牌名，而是底座的工作名，用于文档、架构图和内部讨论。

### 4.3 工作台和入口层命名

| 当前名称 | 推荐目标名称 | 推荐定位 |
|---|---|---|
| `DeanAgent-Backend` | `情报引擎 Core` | 平台底座 |
| `Dean-Agent-Fronted` | `Dean Workbench` 或 `院长决策驾驶舱` | 领导驾驶舱 |
| `Scholars-System` | `Scholar Workbench` 或 `学术生态运营台` | 知识运营台 |
| `academic-monitor` | `Monitoring Center` 或 `监测中心` | 监测能力域 / 工作台 |
| `Dean-NanoBot` | `Agent Gateway` 或 `智能助手入口` | Agent 入口 |

## 5. 推荐方案

## 5.1 稳态收口方案

这是当前最推荐的方案。

| 对象 | 推荐名称 | 说明 |
|---|---|---|
| 平台总名 | `研究院智能情报平台` | 平台总称，适合跨团队统一使用 |
| 底座 | `情报引擎 Core` | 替代 `DeanAgent-Backend` 的定位表达 |
| 领导前台 | `Dean Workbench` | 保留 Dean 场景识别度 |
| 知识运营台 | `Scholar Workbench` | 比 `Scholars-System` 更明确 |
| 监测能力域 | `Monitoring Center` | 同时可表示工作台与能力域 |
| Agent 入口 | `Agent Gateway` | 保持中性，突出入口属性 |

### 优点

- 命名中性且清晰
- 不再混淆平台与 Agent 项目
- 平台叙事更稳定
- 后续扩展更容易

### 缺点

- 需要一段过渡期同步旧名和新名

## 5.2 保守方案

不改仓库名，只改文档和展示层命名。

### 优点

- 成本低

### 缺点

- 仓库层仍保留历史命名负担

## 5.3 不推荐方案

一次性改所有仓库名、部署名、标题名和文档名。

### 原因

- 风险高
- 成本高
- 容易在平台边界尚未完全稳定前造成混乱

---

## 6. 推荐整合顺序

### 第一步：统一文档口径

- 所有规划文档统一使用平台总称和工作名
- README 中增加“当前项目在平台中的角色”

### 第二步：统一展示名称

- 前端标题、介绍页、汇报材料统一使用新命名
- 保留旧名称作为一段时间的兼容说明

### 第三步：统一接口和模型

- 统一 API 口径
- 统一工具口径
- 统一身份映射

### 第四步：评估仓库级调整

- 等平台边界和职责稳定后，再评估是否改仓库名或合并部分仓库

---

## 7. 推荐的逻辑结构

未来逻辑结构建议如下：

```text
Research Intelligence Platform
├── core
├── dean-workbench
├── scholar-workbench
├── monitoring-center
├── agent-gateway
└── shared-docs
```

这不意味着现在必须立刻物理合并，只表示未来结构目标。

---

## 8. `academic-monitor` 的整合命名建议

### 推荐表达

- 作为能力域：`Monitoring Framework`
- 作为工作台：`Monitoring Center`

### 原因

这样可以同时表达：

- 它有底层监测抽象
- 它未来也可以形成面向用户的工作台

### 不建议

- 继续把它长期当作一套平行平台

---

## 9. `Dean-NanoBot` 的命名建议

### 推荐表达

- 对内工作名：`Agent Gateway`
- 对业务方展示名：`智能助手入口`

### 原因

- 弱化历史场景绑定
- 强化它作为统一入口层的角色
- 避免把“入口层”和“平台底座”混淆

---

## 10. 对内与对外两套表达

### 对内研发表达

- `情报引擎 Core`
- `Dean Workbench`
- `Scholar Workbench`
- `Monitoring Center`
- `Agent Gateway`

### 对业务方表达

- 研究院智能情报平台
- 院长决策驾驶舱
- 学术生态运营台
- 目标监测中心
- 智能助手入口

---

## 11. 当前最推荐做法

采用“稳态收口方案”：

- 平台统一叫 `研究院智能情报平台`
- 底座统一叫 `情报引擎 Core`
- 前台按工作台命名
- Agent 入口独立命名为 `Agent Gateway`
- 先统一文档与展示口径，再决定是否修改仓库名

---

## 12. 一句话总结

重命名的目标不是换一批好听的名字，而是让平台边界、产品职责和协作关系变得清晰。
