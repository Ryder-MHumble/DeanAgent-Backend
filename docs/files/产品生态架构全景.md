## 一、平台定位

**研究院智能情报平台** 是中关村人工智能研究院的核心信息基础设施，承担全院的信息感知、数据治理、智能分析三大功能。

平台持续监测 **181 个信息源**，覆盖国家政策、科技前沿、人才动态、产业趋势、高校生态等 **9 个战略维度**，并维护 **2,200+ 位学者**的知识档案和机构关系网络。通过 **65+ REST API 端点**，向院内 4 个上层应用提供统一的数据服务，是院内所有智能应用的数据底座。

## 三、产品生态现状

### 3.1 Dean-Agent — 院长智能体

**服务对象：** 院长办公室、院领导层

**核心场景：**
- 每日政策情报简报，自动提取关键政策机会（资金申报、科研合作）
- 人事变动追踪，监测国家/北京市 AI 领域重要任免动态
- 科技前沿趋势报告，8 大技术主题实时热度与院内应对建议
- 高校动态聚合，一站式了解顶尖高校最新动向

**接入数据：**
```
/api/v1/intel/policy/feed          → 政策动态（规则+LLM富化）
/api/v1/intel/policy/opportunities → 政策机会（资金/申报）
/api/v1/intel/personnel/changes    → 人事变动（结构化提取）
/api/v1/intel/tech-frontier/topics → 科技前沿 8 大主题
/api/v1/intel/daily-briefing/today → 每日简报
/api/v1/intel/university/feed      → 高校动态
```

---

### 3.2 ScholarDB-System — 学者知识库

**服务对象：** 科研管理办公室

**核心场景：**
- 顶尖高校 AI 师资档案管理（姓名/职称/研究方向/论著/荣誉）
- 合作导师与学生关系追踪
- 科研项目从申报到结题全周期管理
- 合作机构与人脉关系图谱维护

**接入数据：**
```
/api/v1/scholars/            → 学者列表（2200+ 人，支持多维过滤）
/api/v1/scholars/{id}/       → 学者详情（含 AMiner 数据补全）
/api/v1/scholars/{id}/students → 导学关系
/api/v1/institutions/        → 机构/院系树（含学者分布统计）
/api/v1/projects/            → 项目库 CRUD
/api/v1/aminer/scholars/search → AMiner 学者检索
```

---

### 3.3 Athena — 战略情报引擎

**服务对象：** 战略发展部

**核心场景：**
- 全量信息检索，跨 9 个维度的政策/产业/技术数据集
- 深度情报分析，支持关键词过滤、时间范围、信源筛选
- 科技机会挖掘，识别产业合作、技术引进、人才引进窗口
- 竞争态势研判，聚焦同类机构动态

**接入数据：**
```
/api/v1/articles/            → 全量文章（138 个信源，支持全文搜索）
/api/v1/articles/stats       → 多维统计分析（按维度/信源/日期）
/api/v1/intel/tech-frontier/ → 科技前沿全套接口
/api/v1/intel/policy/        → 政策情报全套接口
/api/v1/sentiment/           → 社交舆情（Twitter/X KOL 动态）
```

---

### 3.4 NanoBot — 全院统一智能入口

**服务对象：** 全院各部门员工

**核心定位：** 嵌入钉钉的 AI 智能助理，将情报平台的能力统一封装为自然语言交互界面。各部门无需知道具体 API，通过对话即可获取数据和服务。

**核心场景：**
- 「最近有哪些 AI 政策？」→ 调用政策情报 API
- 「帮我查一下清华计算机系的学者列表」→ 调用学者 API
- 「今天的院内简报是什么？」→ 调用每日简报 API
- 「最近有没有国家基金申报机会？」→ 调用政策机会 API

**技术接入方式：**
- MCP（Model Context Protocol）封装情报平台 API
- 通过钉钉机器人 webhook 触发
- 不替代 Dean-Agent/ScholarDB/Athena，而是提供统一的低门槛访问入口

## 四、当前数据治理挑战

坦诚评估现阶段系统的核心局限：

### 挑战 1：存储层缺乏事务保障

现有纯 JSON 文件存储在并发写入时存在数据竞争风险，缺乏原子性保证。当前通过单 Worker APScheduler 和文件锁规避，但制约了横向扩展能力。

### 挑战 2：API 缺乏版本管理与认证

所有 API 端点均为公开访问，无认证鉴权层。4 个消费端共用同一套 API，任何 Breaking Change 均需协调所有消费端同步更新。

### 挑战 3：数据耦合度高

学者数据、文章数据、项目数据共用同一套 Service 层和存储结构，领域边界模糊，难以独立扩展和优化。

### 挑战 4：数据血缘不透明

当前缺乏系统性的数据血缘追踪机制，难以回答「这条数据从哪里来、什么时候更新、准确性如何」等基础问题。

## 五、未来演进方向

基于当前系统现状和院内需求，提出 5 条演进原则（道层面，非具体排期）：

### 方向 1 — 平台化

> 从"爬虫工具集"升级为"院内数据中台"

将情报引擎的定位从功能性工具提升为基础设施平台，建立标准化的数据接入规范、统一的数据模型，让新业务系统能以可预期的方式消费数据，而不是每次都需要定制开发。

### 方向 2 — 数据治理

> 建立分层存储 + 数据质量体系

引入数据分层架构（原始层→清洗层→服务层），配套元数据管理和数据质量监控。让每条数据有清晰的来源追踪、更新频率、质量评分，为上层应用的决策可信度提供保障。

### 方向 3 — 统一接入

> NanoBot 成为院内 AI 应用的统一服务网关

以 MCP 协议为基础，将情报平台的数据能力标准化为可被任何 AI 模型调用的工具集。NanoBot 不仅是一个聊天机器人，更是全院 AI 应用接入数据服务的统一入口，实现"一处接入、全院共享"。

### 方向 4 — 契约优先

> 稳定的 API 契约解耦平台迭代与上层应用

建立正式的 API 版本管理机制（v1/v2 并行），引入认证鉴权层，使各消费端在 API 契约稳定的前提下独立迭代。平台底层可以大幅重构（如引入数据库），但上层应用无感知。

### 方向 5 — 智能驱动

> LLM 从辅助工具升级为核心处理层

当前 LLM 主要用于已有规则引擎无法处理的边缘情况富化。未来考虑将语义理解能力引入更早的处理阶段，如信息去重、自动分类、跨信源实体识别等，从根本上提升数据质量，而不只是靠人工维护规则。
