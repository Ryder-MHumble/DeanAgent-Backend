## 一、框架现状与集成前提

### 1.1 已建立的基础设施（可直接复用）

| 组件 | 位置 | 作用 |
|------|------|------|
| `CrawlerRegistry` | `app/crawlers/registry.py` | 配置 → 爬虫实例的工厂，根据 `crawler_class` 或 `crawl_method` 路由 |
| `BaseCrawler` | `app/crawlers/base.py` | 所有爬虫的抽象基类，统一输出 `CrawledItem` |

### 1.3 数据流（已打通）

```
YAML配置（sources/**/*.yaml）
    ↓
CrawlerRegistry.create_crawler(config)
    ↓
BaseCrawler.fetch_and_parse() → List[CrawledItem]
    ↓
Pipeline Processor（intel/pipeline/*.py）
    ↓
Supabase KG存储（PostgreSQL + 关系型数据）
    ↓
v1 REST API（FastAPI）
    ↓
Dean-Agent / ScholarDB-System / Athena / NanoBot
```

**结论：新信源不是 0-1，是在已有注册-解析-存储链路上插入新节点。**

#### B. 顶会高质量论文作者（4个信源）

| # | 来源 | 真实URL | API支持 | 爬取方式 | 难度 | 备注 |
|---|------|---------|---------|----------|------|------|
| 12 | **DBLP** | `https://api.dblp.org/` | ✅ 官方API（XML/JSON） | `https://api.dblp.org/search/authors?q={query}` | ⬛ **最低** | **首选**。官方API稳定，按作者名/机构检索，返回 PID（唯一标识） |
| 13 | **Semantic Scholar** | `https://api.semanticscholar.org/graph/v1/` | ✅ 官方API | `GET /author/search?query={name}` | 🟡 中 | 已有 `semantic_scholar` parser，但目前是 Paper 检索，需扩展作者检索；需要 API Key |
| 14 | **arXiv** | `http://export.arxiv.org/api/query?search_query=au:` | ✅ 官方API | `http://export.arxiv.org/api/query?search_query=au:{author}&max_results=100` | ⬛ **低** | **已有 ArxivAPICrawler**，直接启用即可 |
| 15 | **OpenReview** | `https://api.openreview.net/` | ⚠️ 非官方API | `GET /profiles?email={email}` | 🔴 高 | 需账号token，反爬较严；建议仅作为辅助验证 |

> **推荐数据获取优先级：DBLP > arXiv > Semantic Scholar > OpenReview。DBLP 的 PID 是作者消歧的最佳方案。**

#### C. GitHub 主要贡献者（3个信源）

| # | 来源 | 真实URL | API支持 | 爬取方式 | 难度 | 备注 |
|---|------|---------|---------|----------|------|------|
| 16 | **GitHub Search Users** | `https://api.github.com/search/users` | ✅ 官方API | `GET /search/users?q=followers:>100+repos:>20+language:Python&per_page=100` | ⬛ **低** | 搜AI领域高活跃用户；username → API查详情（bio/location/email） |
| 17 | **GitHub Repo Contributors** | `https://api.github.com/repos/{owner}/{repo}/contributors` | ✅ 官方API | 遍历目标仓库列表 | 🟡 中 | 需要先定义目标AI仓库列表；可用 stars/topics 筛选 |
| 18 | **GitHub User Profile** | `https://api.github.com/users/{username}` | ✅ 官方API | 用户详情页（bio/company/location/email） | ⬛ **低** | GitHub email 公开率低（约5%），需结合其他信息补充真实身份 |

> **身份关联难点**：GitHub username ≠ 真实姓名 ≠ 学校。解决方案：bio中提取 → GitHub Education → 交叉论文DBLP数据。

### 4.2 DBLP 作者检索 Parser

```python
# app/crawlers/parsers/dblp_author.py

class DBLPAuthorCrawler(BaseCrawler):
    """
    DBLP 官方API - 按作者名/机构检索顶会论文作者

    数据流：
      1. GET https://api.dblp.org/search/authors?q={query}&format=json
         → 获取 authors[]（含 author_pid）
      2. 对每个 author_pid:
         GET https://api.dblp.org/author/{pid}?format=json
         → 获取作者详情（姓名、机构、论文列表）
      3. 过滤顶会范围：ICML, ICLR, NeurIPS, AAAI, ICCV, ACL, CVPR, EMNLP
      4. 按 paper 数量 + citation 排序，取主要作者

    Config fields:
      - search_query: str，作者名或机构关键词
      - venues: list[str]，顶会过滤范围，默认 ["ICML", "ICLR", "NeurIPS", "AAAI", "ICCV", "ACL", "CVPR", "EMNLP"]
      - max_results: int，默认 50
      - use_affiliation_filter: bool，是否只保留中国高校作者

    输出字段：
      - title: 作者姓名
      - url: DBLP 个人主页
      - author: 机构
      - content: 个人简介/研究领域
      - extra.dblp_pid: 作者唯一标识符
      - extra.paper_count: 论文总数
      - extra.venue_papers: {venue: count} 各顶会论文数
      - extra.affiliation: 机构
      - extra.homepage: 作者个人主页
    """
    pass
```

### 4.3 GitHub 贡献者 Parser

```python
# app/crawlers/parsers/github_contributor.py

class GitHubContributorCrawler(BaseCrawler):
    """
    GitHub 用户搜索 + 仓库贡献者爬虫

    两条数据路径：

    路径1 - 用户搜索（推荐）：
      GET https://api.github.com/search/users
        ?q=language:Python+followers:>100+repos:>10&per_page=100&sort=followers
      → 遍历每个 user:
        GET https://api.github.com/users/{username}
        → 获取 bio/company/location/email/blog

    路径2 - 仓库贡献者（精准）：
      先通过 GitHubAPI 搜 AI 相关高星仓库：
        GET https://api.github.com/search/repositories
          ?q=topic:artificial-intelligence+stars:>1000&sort=stars&per_page=50
      再对每个仓库：
        GET https://api.github.com/repos/{owner}/{repo}/contributors
        ?per_page=100
      → 获取贡献者列表（username, contributions_count）

    Config fields:
      - search_type: "users" | "repo_contributors"，默认 "users"
      - target_repos: list[str]，目标仓库列表（如 ["microsoft/DeepSeek", "THUDM/ChatGLM3"]）
      - min_followers: int，默认 100
      - min_repos: int，默认 10
      - language_filter: str，默认 "Python"

    输出字段：
      - title: GitHub username
      - url: GitHub 个人主页
      - author: 真实姓名（如从 bio 提取到）
      - content: bio 简介
      - extra.github_id: 用户数字ID
      - extra.followers: 粉丝数
      - extra.public_repos: 公开仓库数
      - extra.company: 公司/组织
      - extra.location: 地理位置
      - extra.blog: 个人网站
      - extra.email: 邮箱（极少数公开）
      - extra.contribution_count: 总贡献次数
    """
    pass
```

## 五、Pipeline 扩展

### 5.1 新 Processor 文件

```python
# app/services/intel/pipeline/student_profile_processor.py

class StudentProfileProcessor:
    """
    学生画像数据处理Pipeline

    职责：
    1. 接收 CrawledItem 列表（来自各Parser）
    2. 按数据类型（competition_winner / paper_author / github_contributor）分类
    3. 去重（基于 content_hash 或 url_hash）
    4. 身份关联消歧：
       - GitHub username → 与 DBLP author 交叉匹配姓名
       - 同一姓名多来源 → 建立统一学生档案
    5. 输出 KG 格式写入 Supabase：
       - Node: StudentProfile {id, name, university, email, sources[]}
       - Relation: StudentProfile -[AWARDED_IN]-> Competition
       - Relation: StudentProfile -[AUTHORED_PAPER_IN]-> Venue
       - Relation: StudentProfile -[CONTRIBUTED_TO]-> Repository

    API 输出格式（/api/v1/student-profile）：
      {
        "generated_at": "2026-04-24T12:00:00Z",
        "item_count": 150,
        "items": [
          {
            "id": "sp_001",
            "name": "张三",
            "university": "清华大学",
            "department": "计算机科学与技术系",
            "email": "zhangsan@tsinghua.edu.cn",
            "sources": ["ccf_bdci", "dblp_author"],
            "tags": ["competition_winner", "paper_author"],
            "extra": {
              "awards": [{"competition": "CCF BDCI 2024", "rank": "一等奖", "year": 2024}],
              "papers": [{"venue": "ACL 2024", "title": "...", "citation": 25}],
              "github": {"username": "zhangsan-a", "repos": 23, "contributions": 1842}
            }
          }
        ]
      }
      """
    pass
```

## 八、风险与对策

| 风险 | 级别 | 说明 | 应对 |
|------|------|------|------|
| **GitHub API限速** | 🟡 中 | 未认证60次/时，认证5000次/时 | 在`.env`配置`GITHUB_TOKEN`；每个Parser加`request_delay` |
| **作者消歧** | 🔴 高 | DBLP/arxiv/Semantic Scholar 同名作者需要消歧 | DBLP PID作为唯一标识；建立 `name → DBLP_pid` 映射表；结合机构/论文共现 |
| **天池/BDCI反爬** | 🟡 中 | 工业平台通常有反爬 | 配`request_delay: 2`，启用`use_playwright: true` |
| **数据新鲜度** | 🟡 中 | 竞赛获奖者名单实时性弱 | 设`schedule: weekly`；首期人工维护获奖者列表后增量追踪 |
| **GitHub→真实身份** | 🔴 高 | username ≠ 真实姓名 ≠ 学校 | bio提取 → GitHub Education → 交叉论文DBLP数据（最高优先级） |
| **OpenReview需账号** | 🟡 中 | 非官方API，需注册账号token | Phase 1 先禁用，Phase 2 视需求决定是否接入 |
| **Supabase KG Schema** | 🟡 中 | 需要新增学生画像的Node/Relation类型 | 在Supabase中新建`student_profiles`表和关联表 |
| **Kaggle Playwright** | 🟡 中 | React SPA，静态HTML无数据 | 已有`use_playwright: true`方案，需解决登录验证码 |
