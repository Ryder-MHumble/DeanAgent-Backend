# API v2 准入评估清单

> 目标：为 `/api` 的演进提供统一、可量化的“是否进入 v2”决策标准。
> 适用范围：所有可能影响现有消费端（Dean-Agent / ScholarDB / Athena / NanoBot）的 API 变更评审。

## 1. 判定原则

默认策略：**能兼容就留在 v1**。
仅当出现“系统性破坏兼容”或“兼容成本倒挂”时，进入 v2。

`non-breaking`（通常保留 v1）：
- 新增端点
- 新增可选参数
- 新增可选响应字段
- 性能优化、内部重构（外部契约不变）

`breaking`（进入 v2 候选）：
- 路径或 HTTP 方法变化（如 `GET /x` 改为 `POST /y`）
- 字段语义变化（同名字段含义变化）
- 必填参数新增/重命名/删除
- 响应结构重组（分页模型、字段层级、错误码体系变化）
- 状态码行为改变（例如原 `200` 改为 `202/4xx`）

## 2. 变更登记表（逐条填写）

| 端点 | 变更内容 | 类型 | 影响消费端 | 预计改造量（人天） | 可否兼容实现 |
|---|---|---|---|---:|---|
| `GET /api/xxx` | 示例：`status` 字段语义调整 | `breaking` | Dean-Agent, Athena | 2 | 否 |
| `POST /api/yyy` | 示例：新增可选参数 `source` | `non-breaking` | 无 | 0 | 是 |

## 3. v2 准入评分卡

评审时对本次“计划变更集合”打分。

| 指标 | 计算方式 | 触发条件 | 分值 |
|---|---|---|---:|
| Breaking 端点占比 | `breaking端点数 / 计划变更端点数` | `>= 20%` | 30 |
| 高频端点破坏数 | Top 20% 流量端点中的 `breaking` 数量 | `>= 3` | 25 |
| 消费端影响范围 | 受影响客户端数量 | `>= 2` | 20 |
| 兼容补丁复杂度 | 为维持 v1 需新增的分支/适配层复杂度 | 高 | 15 |
| 语义一致性风险 | 参数命名、分页、错误码是否已明显割裂 | 高 | 10 |

总分判定：
- `>= 60`：进入 `v2` 立项
- `40 ~ 59`：暂不立项，冻结高风险变更并做兼容方案复核
- `< 40`：继续在 `v1` 迭代

## 4. 立项结论模板（会议纪要）

| 项目 | 内容 |
|---|---|
| 评审日期 | YYYY-MM-DD |
| 结论 | `进入v2` / `继续v1` |
| 评分结果 | XX 分 |
| v1 策略 | 仅 bugfix / 安全补丁 / 小幅 non-breaking |
| v2 目标 | 例如：统一分页模型、统一错误码、重构机构领域契约 |
| 并行周期 | 例如：2~3 个发布周期 |
| v1 EOL 日期 | YYYY-MM-DD |
| Owner | 负责人姓名 |

## 5. v2 启动后的最小交付标准

1. `v1` 与 `v2` 并行发布，避免强制一次性迁移。
2. 提供端点映射表：`v1 -> v2`（含参数映射）。
3. 每个 `breaking` 变更提供迁移示例（请求/响应）。
4. 发布前完成 4 个消费端联调确认。
5. 在文档中明确弃用时间线（公告日、冻结日、下线日）。

## 6. 建议评审流程（每次大版本前）

1. 收集候选变更，填写“变更登记表”。
2. 架构组与客户端代表共同标记 `breaking/non-breaking`。
3. 计算评分卡并给出结论。
4. 若进入 `v2`：同步输出迁移计划与时间线。
5. 若继续 `v1`：记录兼容实现方案和风险兜底。

---

## 附录 A：快速判定问题（5 选 1 以上为“高风险”）

- 是否有路径/方法变化？
- 是否有字段语义变化？
- 是否有必填参数调整？
- 是否影响 2 个以上消费端？
- 是否需要大量兼容分支才能保持 v1 契约？

若以上任意项为“是”，建议进入评分卡流程，不要直接拍脑袋决定。
